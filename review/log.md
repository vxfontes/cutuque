## 2026-07-03 — histórico ao adotar, sessões LIVE no Mac, supressão de push por foreground, preview + markdown
**Risco:** R2 (Alto — novos endpoints públicos (`/machines/{m}/live`, `/app/foreground`), importação de histórico de conversa via SSH para máquinas remotas, e uma mudança em lógica de segurança de notificação (supressão de push). Sem RCE novo — SEC-101 não regrediu.)
**Tipo:** T1 (feature nova) + T3 (refactor: `discoveryLikeHandler` compartilhado)
**Escopo:** hub: `adapter/claudecode/{transcript.go (novo), live.go (novo), discover.go, target.go}`, `launcher/launcher.go` (`Adopt`/`importTranscript`/`Live`), `registry/output.go` (teto 200→500), `notifier/notifier.go` (`SetForeground`/TTL), `server/{settings.go (novo), launch.go, server.go}`, `session/session.go`, `cmd/hub/main.go`. app: `MarkdownText.swift` (novo), `ForegroundReporter.swift` (novo), `APIClient.swift`, `Models.swift`, `CutuqueApp.swift`, `DiscoverSessionsView.swift`, `SessionListView.swift`, `SessionDetailView.swift`.

### Bloqueantes
1. **issue (correctness/concurrency, blocking, impacto de segurança):** corrida entre o heartbeat de foreground (`ForegroundReporter.swift:36-43`, manda `active:true` a cada 60s) e o aviso de background (`ForegroundReporter.swift:45-50`, manda `active:false` na hora) — são duas requisições HTTP independentes, sem nenhuma ordenação/sequência entre elas. `Notifier.SetForeground` (`notifier.go:92-98`) só faz `atomic.Int64.Store`, que garante atomicidade do acesso à variável, mas NÃO garante que as chamadas sejam aplicadas na ordem em que os eventos aconteceram no cliente — só na ordem de CHEGADA ao hub. Cenário concreto: qualquer transição para `.inactive` (Control Center, app switcher, chamada entrando, Face ID) já dispara `stopHeartbeat()` → `active:false` imediato; ao voltar para `.active`, `startHeartbeat()` dispara `active:true` imediato, sem debounce. Duas requests saem em sequência rápida sem esperar a anterior terminar; se a rede entregar fora de ordem (ou a `true` simplesmente responder mais devagar que uma `false` mandada depois), o hub fica com `foregroundUntil` setado para +150s mesmo com o app em background — suprimindo silenciosamente pushes de `needs_you`/`done`/`error` por até 150s exatamente quando ninguém está olhando o app. Não é um edge case raro: qualquer gesto de Control Center/app switcher já é suficiente para gerar o par de requests concorrentes.
   - **Por que importa:** o objetivo declarado da feature #3 é "nunca perder o push crítico quando o app não está aberto"; essa corrida inverte exatamente essa garantia.
   - **Sugestão concreta:** incluir um timestamp/sequência monotônica no corpo (`{"active": bool, "at": <ms epoch do cliente>}`) e o hub só aplicar a atualização se `at` for mais recente que a última aceita (`notifier.go` guarda `lastForegroundAt int64` ao lado de `foregroundUntil`, ignora updates com `at` mais antigo). Resolve o problema de raiz (ordem lógica, não ordem de chegada na rede) sem exigir sincronização perfeita no cliente.
   - `[→ security.md#SEC-102]`

2. **issue (correctness, blocking):** teto de chunks de output no cliente ficou desatualizado em relação ao hub. `registry/output.go:7` subiu `maxOutputChunks` de 200 para 500 justamente para caber o histórico importado (`Launcher.Adopt` → `importTranscript`). O app, porém, ainda usa um teto fixo de 200 só no caminho de chunk AO VIVO (`SessionDetailView.swift:55-56`, comentário "espelhando o teto de 200 do hub" — não é mais verdade) e NENHUM teto no carregamento inicial (`SessionDetailView.swift:29-30`, `chunks = history` sem `.suffix`). Resultado: ao adotar uma sessão com histórico grande (até 500 chunks importados) e continuar a conversa, o PRIMEIRO chunk ao vivo que chega dispara `removeFirst(count - 200)` (linha 56) e descarta de uma vez até ~300 mensagens do histórico que a usuária acabou de importar — bem no momento em que ela está engajada na conversa. Reprodutível de forma determinística (não depende de corrida), sempre que o histórico importado exceder 200 chunks.
   - **Por que importa:** anula na prática o valor da feature #1 (importar histórico) assim que a sessão volta a ficar viva.
   - **Sugestão concreta:** subir o teto do cliente para 500 (idealmente uma constante única referenciada nos dois lados, ou documentada em ambos os comentários) e aplicar o MESMO teto também no carregamento inicial (`chunks = Array(history.suffix(500))`), não só no caminho de append ao vivo.

3. **issue (concurrency, blocking — 3ª ocorrência de um padrão já catalogado):** `Launcher.Adopt` (`launcher.go:262-283`) faz `reg.Get(id)` e, mais adiante, `reg.Add(s)` + `importTranscript` sem uma seção crítica única cobrindo check+ação — mesma classe de bug do padrão `reivindicação-não-atômica-de-recurso-compartilhado` já catalogado 2x em `hub/review/patterns.md` (ambas as ocorrências anteriores corrigidas). Na revisão da feature de discover/adopt (2026-07-03) isso já tinha sido apontado como non-blocking, porque um `Add` duplicado do mesmo `Session{}` struct é inofensivo (idempotente). Esta leva muda o cálculo: agora `importTranscript` roda incondicionalmente após o `Add`, e `Registry.AppendOutput` (`registry/output.go:27`) só faz `append` puro, sem chave de idempotência. Duas chamadas concorrentes de `Adopt` para o MESMO id novo (plausível com o app aberto em dois dispositivos apontando pro mesmo hub, ou um retry de cliente) passam ambas pelo `Get` (ambas veem `ok=false`), ambas chamam `importTranscript`, e o histórico inteiro da sessão aparece DUPLICADO no chat.
   - **Por que importa:** o mesmo padrão já mordeu o projeto 2x antes (`Handle.Close`, `respond()`); desta vez o efeito é visível para a usuária (mensagens repetidas), não mais "inofensivo".
   - **Sugestão concreta:** `Registry.AddIfAbsent(s) (existing session.Session, added bool)` fazendo lookup+insert sob o mesmo lock; `Adopt` só chama `importTranscript` quando `added == true`.
   - `[→ hub/review/patterns.md#reivindicação-não-atômica-de-recurso-compartilhado]`

### Non-bloqueantes
- suggestion (perf, if-minor): `importTranscript` roda de forma síncrona dentro do handler de `POST /adopt`, com teto de 15s (`launcher.go:295`, `discoverTimeout`). É uma escolha deliberada e documentada (`launcher.go:279-281`: garante que o `GET /output` logo após o adopt já tenha histórico) e o teto evita travar para sempre — aceitável hoje. Vale reconsiderar para assíncrono (201 imediato + import em background + WS) se popular sessões com transcript grande via SSH remoto se mostrar perceptivelmente lento na prática.
- suggestion (design, recurring): execução de script remoto via SSH sem login shell (`ssh <opts> -- dest "python3 -"`) aparece agora em 3 lugares (`discover.go:104`, `live.go:116`, `transcript.go:125`) com o mesmo risco de PATH restrito em sessão ssh não-interativa já apontado (non-blocking) para `discover.go` na revisão anterior. Vale corrigir uma vez só no nível do helper compartilhado (`runDiscoverScript`/equivalente para `runTranscript`) em vez de manter a mesma decisão replicada em 3 arquivos. `[→ review/patterns.md#ssh-remote-script-sem-login-shell]`
- suggestion (correctness/UX, fail-open, baixa severidade): `CutuqueApp.swift:23-24` — `.onChange(of: scenePhase)` não usa `initial: true`, então `ForegroundReporter.shared.update(.active)` nunca roda no cold launch; o hub só aprende "foreground=true" na primeira transição background→active. Entre o cold launch e essa primeira transição, um push de `needs_you` pode disparar mesmo com o app aberto — direção de falha é "notifica a mais", não "suprime indevidamente", por isso non-blocking.
- thought (design): heurística `newest_sid(cwd)` do `liveScript` (`live.go:68-71`) mapeia um cwd para UM candidato (arquivo `.jsonl` de mtime mais recente). Duas sessões `claude` vivas simultaneamente no MESMO cwd (ex.: duas abas de terminal no mesmo projeto) colapsam no mesmo candidato — a outra sessão viva fica invisível naquele poll e pode "piscar" entre polls de 15s conforme a ordem de escrita dos dois processos. Não é crash nem segurança; já é um fallback (o sinal forte é `--session-id`/`.jsonl` no argv, `live.go:62-67`). Vale documentar a limitação ou, se incomodar na prática, trocar por `lsof -p <pid>` filtrando descritores `.jsonl` abertos de fato pelo processo, em vez de cwd+mtime.
- nitpick: `sid_from_cmd` (`live.go:62-67`) reconhece `--session-id <uuid>` e `<uuid>.jsonl` no argv, mas não `--resume <uuid>` — que é exatamente a flag usada pelo próprio `Launcher.resume` (`launcher.go:425`) para retomar sessões adotadas/continuadas pelo Cutuque. Na prática cai no fallback de cwd sem problema visível (o arquivo da sessão retomada quase sempre é o mais recente no seu próprio cwd), mas um `--resume\s+(UUID)` a mais no regex tornaria o match forte também para o próprio tráfego do hub.
- nitpick (perf, irrelevante na escala atual): `importTranscript` chama `AppendOutput` uma vez por chunk em loop (até 500 vezes), cada chamada faz lock/unlock + tentativa de broadcast (`registry/output.go:27-41`) — para uma sessão recém-adotada isso é uma rajada de até 500 eventos `output_chunk` na conexão WS geral (que ninguém ainda está filtrando para essa sessão, então são descartados sem efeito). Não é bug (o `Output(id)` usado pelo `GET /output` lê o array persistido, não depende do broadcast), só uma rajada desnecessária de mensagens.

### Confirmações positivas (respondendo aos pontos específicos pedidos)
- **Ponto #6 (SEC-101):** NÃO regrediu. `SSHTarget.Transcript` (`transcript.go:124-127`) reusa `singleQuote`/um único nível de shell (sem `bash -lc` aninhado) e depende de `Launcher.Adopt` validar `id` contra `sessionIDPattern` (`launcher.go:259-261`, `^[0-9a-fA-F-]{8,64}$`) antes de qualquer chamada — único call site confirmado via grep. `Live`/`live.go` não recebe nenhum input do cliente (script é uma constante; `machine` só é usado para indexar o mapa fixo de targets).
- **Ponto #2:** `importTranscript` bloqueia o handler de `/adopt` por até 15s no pior caso, mas com teto (`discoverTimeout`) e degradação graciosa (erro → retorna silenciosamente, sessão fica adotada sem histórico). Ver suggestion non-blocking acima.
- **Ponto #4:** `livePollTask` (`SessionListView.swift:88-102`) não vaza — `guard let self else { return }` está DENTRO do `while` (reavaliado a cada iteração), diferente do bug já catalogado `weak-self-fora-do-loop-em-task-de-stream`; `stopLivePolling()` cancela e zera a referência, chamado em `.onDisappear`. Classe é `@MainActor`, então toda escrita em `@Published` (`refreshLive`, `liveSessions =`) roda isolada na main actor — nenhuma escrita fora da main encontrada.
- **Ponto #5:** `MarkdownText.swift` não apresenta loop infinito nem crash nos casos testados (fence sem fechamento, heading sem espaço, tabela malformada, índices fora de faixa) — todo `while` interno usa `i < lines.count` como curto-circuito antes de indexar, e o parser inline cai em `Text(s)` puro via `try?` quando `AttributedString(markdown:)` falha.
- **Ponto #1:** `liveScript` não usa `shell=True` em nenhum subprocess (`ps`/`lsof` chamados com lista de args), e o script é uma constante Go sem interpolação de input externo — sem risco de injeção. Risco de misattribution é real mas limitado (ver thought/nitpick acima), mitigado pela janela de recência de 180s.

## 2026-07-03 — feature "descobrir e adotar sessões ativas do Claude Code no Mac"
**Risco:** R1 (novo endpoint que aceita `id`/`cwd` do cliente e os injeta num `bash -lc` remoto via SSH — RCE confirmado por PoC)
**Escopo:** hub/internal/adapter/claudecode/{discover.go (novo), target.go}, hub/internal/launcher/launcher.go (Discover/Adopt/resume), hub/internal/session/session.go, hub/internal/server/{launch.go,server.go}, app/CutuqueApp/{APIClient.swift, Models.swift, DiscoverSessionsView.swift (novo), SessionListView.swift}

### Bloqueantes
- issue (security, blocking): OS command injection via `id` do Adopt no caminho SSH — `remoteClaudeCommand` junta `resumeID` numa string que é reparseada por um `bash -lc` ANINHADO (double-shell), então `singleQuote` só protege o nível externo. PoC local confirmou RCE (`; touch <arquivo>` executado). [→ security.md#SEC-101]
- issue (testability, blocking): `hub/internal/adapter/claudecode/discover.go` é um arquivo NOVO inteiro sem nenhum teste (`discover_test.go` não existe) — `parseDiscovered`, `runDiscover` e os dois `Discover` não têm cobertura alguma; um teste de integração com fake ssh (no espírito de `TestSSHTargetRunnerProcessesFixtureViaFakeProgram`) teria pego o achado acima.
- issue (correctness, blocking): `DiscoverHandler`/`Launcher.Discover` mapeiam QUALQUER erro (ssh caiu, python3 ausente, script crashou, JSON corrompido) para 404 `unknown_machine`, indistinguível de "máquina não configurada". Quebra o padrão já usado em `LaunchHandler` (dispatch por `errors.Is`).

### Non-bloqueantes
- suggestion (perf/reliability): `Discover` roda sobre `l.baseCtx` sem timeout próprio (diferente de `launchTimeout` do Launch) — uma ssh/python3 pendurada trava o request e o processo até o Shutdown do hub.
- suggestion (concurrency, if-minor): `Adopt` faz `reg.Get` + `reg.Add` sem lock atômico (TOCTOU) — inconsistente com o cuidado explícito de `respond()` (reivindicação atômica do pending). Risco prático baixo (exige colisão de id), mas vale um `reg.AddIfAbsent`.
- suggestion: `SSHTarget.Discover` roda `python3 -` direto (sem `bash -lc`), diferente do padrão usado pro `claude` — pode falhar em máquinas com PATH restrito na sessão ssh não-interativa.
- suggestion (UX): `APIClient.discover` engole QUALQUER erro em `[]`, indistinguível de "sem sessões" — diferente de `adopt`, que propaga erro pra `errorMessage`.
- nitpick: `discoverScript` não colapsa espaços/quebras de linha do título (`title=c.strip()[:100]`) — títulos multi-linha ficam poluídos na lista.
- thought: comentário de `singleQuote` ("não pretende ser escapador genérico para input externo") ficou desatualizado — é exatamente o que a feature faz agora; vale atualizar junto do fix.
- praise: `parseDiscovered` + o script python tratam saída vazia/arquivo corrompido com graça (nunca quebra a lista inteira por causa de uma sessão ruim) — bom design defensivo.
- praise: no app, `DiscoverSessionsView` desabilita TODAS as linhas durante uma adoção em voo e `SessionListView` guarda contra push duplicado no path (`if path.last?.id != session.id`) — detalhe de concorrência de UI bem pensado.
