### Bloqueantes
1. **issue (correctness, blocking):** o comentário novo em `notifier.go:196-205` afirma "um push por tarefa concluída, não por turno interno" — essa invariante é FALSA para qualquer sessão usada interativamente (exatamente o caso de sessões `External`/tmux, que é o alvo desta correção). Trace do ciclo completo: `UserPromptSubmit→running` (sem push) → ... → `Stop→done` (push #1, correto) → `Notification` ocioso 60s depois → `Finished` de novo, mas `Apply` já faz no-op porque `cur.State == target` (`engine.go:68-69`) — sem push, correto, é o que os testes novos cobrem. MAS: se a usuária mandar uma NOVA mensagem na mesma sessão tmux (`UserPromptSubmit→running`, transição real, `n.states[id]` grava `running`) e o Claude responder (`Stop→done`, transição real de `running` para `done`) — isso passa no dedup de `handle()` (`prev=running != done`) e dispara PUSH #2. Cada turno subsequente da MESMA conversa interativa repete esse ciclo e gera OUTRO push "concluiu". Para uma sessão de pareamento longa no terminal (o caso de uso que `External` existe para servir) isso é um push por resposta do Claude, não um push pela tarefa toda — exatamente o "risco real de push repetido" que a revisão pediu para escrutinar, e o oposto do que o comentário reivindica.
   - **Por que importa:** é o comportamento que esta leva inteira se propõe a consertar (ruído de notificação em sessões tmux/externas); ele permanece para o padrão de uso mais comum de uma sessão `External` (conversa interativa de ida-e-volta), só que agora documentado como resolvido — o que pode impedir alguém de notar/reabrir o problema depois.
   - **Sugestão concreta:** não deixar o `Stop` isolado disparar PUSH diretamente para sessões que continuam vivas para responder; usar a Notification ociosa ("waiting for your input", já classificada corretamente por `notificationBlocks`) como o sinal real de "ninguém está mais dirigindo isso agora", e tratar `Stop` como só uma atualização de estado da UI (pode virar `done` no Registry imediatamente, ok) sem necessariamente dela sozinha disparar a chamada de `buildPush`+`fanout`. Alternativa mais simples e menos invasiva: um cooldown mínimo por sessão em `Notifier.handle` (ex.: não repetir push de `done` para o mesmo id dentro de N segundos do último `done` empurrado), que amortece o caso de conversa acelerada sem mexer na máquina de estados.
   - **Observação relacionada (não confirmada com teste, só leitura de código/documentação pública do SDK):** o mesmo padrão de `Finished` por `"result"` de CADA turno (`parser.go: resultEvent`) provavelmente também afeta sessões NÃO-`External` mantidas vivas via `SendText` no mesmo handle (`launcher.go`) — o `Runner.Run` não distingue "resultado de turno" de "resultado de tarefa" nesse caminho também. Vale um teste de fixture com duas linhas `result` em sequência no mesmo stream para confirmar antes de decidir o escopo do fix.

2. **issue (correctness/concurrency, blocking — 5ª ocorrência do padrão `reivindicação-não-atômica-de-recurso-compartilhado`):** o fix do bloqueante #2 da leva anterior trocou `EnsureRegistered` para usar `AddIfAllowed` (atômico) — correto — mas `Engine.ensureRunning` (`engine.go:86-102`), o OUTRO caminho que também cria sessão do zero (usado tanto pelo `Runner` quanto pelo dispatch de hook `SessionStart`/`UserPromptSubmit`), continua fazendo `Get` seguido de `Add` bruto como duas aquisições de lock separadas, em vez de `AddIfAbsent` (já disponível e usado em `Adopt`/`AddIfAllowed`). Cenário: hook e Runner recebem o `SessionStarted` de uma sessão NUNCA vista por nenhum dos dois, quase simultaneamente (exatamente a premissa do SEC-103 original). Se o `Get` de um dos dois lados retornar `exists=false` e, ANTES do seu `Add` bruto rodar, o outro lado tiver inserido E JÁ AVANÇADO o estado (ex.: o hook processou seu próprio `EnsureRegistered`+`Apply(SessionStarted)` e, em seguida, um `Notification` de permissão real chegou primeiro, levando a sessão a `needs_you` com `PendingPrompt` preenchido) — o `Add` bruto do outro lado SUBSTITUI o registro inteiro por um `session.Session{}` novo em `StateRunning`, apagando silenciosamente o `needs_you`/`PendingPrompt`/`Pane` que já existiam. Diferente do SEC-103 original (sessão fica visivelmente travada em `needs_you` sem botão), aqui o resultado é PIOR em silêncio: a sessão volta a aparecer como `running`, sem badge `needs_you` nenhum, enquanto o processo `claude` real continua bloqueado esperando resposta no stdin — nenhum sinal visível de que algo está errado. A janela é mais estreita que a do SEC-103 original (exige 3 eventos entrelaçados, não 2), mas o dano pior compensa a menor probabilidade.
   - **Por que importa:** reabre a mesma classe de corrida que acabou de ser fechada em `EnsureRegistered`, por um caminho irmão que faz exatamente a mesma coisa (criar sessão do zero) sem a mesma primitiva atômica.
   - **Sugestão concreta:** trocar o branch de criação de `ensureRunning` para usar `e.reg.AddIfAbsent(novaSessao)`; se `added==false`, cair no mesmo bloco de reconciliação (`Reclaim`/`UpdateState`/`SetPane`) já existente para o caso "já existia" — mesmo padrão que fechou o bloqueante #2 anterior.
   - `[→ review/patterns.md#reivindicação-não-atômica-de-recurso-compartilhado]` `[→ security.md#SEC-106]`

3. **issue (correctness, blocking):** a correção do hijack de pane same-machine (bloqueante #3 da leva anterior, `Registry.SetPane`, `registry.go:189-230`) fechou o caso reportado mas introduziu uma lacuna nova: o loop de eviction (`registry.go:206-217`, `if oid != id && os.Pane == pane`) compara SÓ a string do pane, sem checar `os.Machine == s.Machine`. O alvo composto (`"<socket>\t<pane>"`, montado em `hookPayload.paneTarget()`, `hooks.go:32-43`) não inclui a máquina em NENHUM lugar do código (hub ou app). Como tmux usa por padrão o socket "default" e a primeira pane sempre é "%0", duas máquinas diferentes rodando uma sessão tmux default cada uma colidem na MESMA string de pane com bastante probabilidade na prática (não é um edge case exótico). Consequência no hub: uma sessão `needs_you` na Máquina A pode ser evictada (pane limpo, estado forçado para `done`) por uma sessão nova que só coincide na string de pane na Máquina B — dado de OUTRO usuário/máquina interferindo silenciosamente no estado de uma sessão não relacionada. No app, o mesmo problema aparece em `SessionListView.swift`: `livePaneIDs`/`needsYouPaneIDs` (linhas 202, 204) são `Set<String>` construídos a partir de TODAS as máquinas configuradas (`targets`, poll de `/machines/{m}/live`) sem incluir `machine` na chave, e `liveState(_:)` (linha 435-436) faz `model.sessions.first(where: { $0.tmuxTarget == entry.id })` — sem filtrar por máquina, então uma linha "Ao vivo" da Máquina B pode herdar o estado/dedup de uma sessão da Máquina A que colide na string de pane.
   - **Por que importa:** o mesmo bug de "roubo de identidade de sessão pela pane" que a leva anterior corrigiu para o caso same-machine reaparece para o caso cross-machine, exatamente a configuração multi-máquina que o Cutuque já suporta nativamente (`Launcher.targets`/`SSHTarget`).
   - **Sugestão concreta:** patch mínimo imediato — adicionar `&& os.Machine == s.Machine` à condição de eviction em `SetPane`. Fix durável — incluir `machine` na própria chave composta (`"<machine>\t<socket>\t<pane>"`) em TODO lugar onde ela é construída/comparada (`hooks.go: paneTarget()`, `registry.go: SetPane`, e no app: `LiveEntry`/`tmuxTarget`/dedup sets), para que qualquer código futuro que compare panes não precise lembrar de checar `machine` separadamente — a ambiguidade fica embutida na string hoje, o que é frágil.
   - `[→ security.md#SEC-104]`

4. **issue (correctness/concurrency, blocking):** `devices/store.go: persist()` (linhas 78-91) é chamado FORA do lock de mutação em `Upsert`/`Remove`, e nada serializa chamadas concorrentes de `persist()` entre si — o `path+".tmp"` é um caminho COMPARTILHADO entre todas as chamadas. Duas goroutines concorrentes (ex.: um `Upsert` do app reabrindo, concorrente com um `Remove` disparado por um 410 Unregistered no fanout de push) podem interlaçar leitura de `List()` + `json.Marshal` + `os.WriteFile(tmp,...)` + `os.Rename` de forma que a chamada que TERMINA por último vence, independente de qual mutação era logicamente mais recente — o arquivo em disco pode silenciosamente reverter para um snapshot mais antigo do que o estado em memória (`byToken`, que É protegido corretamente por `sync.RWMutex` e nunca fica inconsistente sozinho). Não é pego por `go test -race` (é uma corrida de I/O de arquivo do SO entre duas chamadas de `os.WriteFile`/`os.Rename`, invisível ao detector de raça do Go, que só instrumenta memória gerenciada pelo runtime) — coerente com o fato de a suíte inteira passar com `-race` sem detectar isto.
   - **Por que importa:** um device removido (410) pode "voltar" no disco se seu `Remove` perder a corrida contra um `Upsert` mais antigo que ainda não tinha terminado de persistir — na prática, reenviar push para um token morto (inofensivo pro usuário, mas desperdiça budget de APNs e pode gerar ruído de log) ou, no sentido oposto, perder um `Upsert` recém-chegado após restart do hub.
   - **Sugestão concreta:** um `persistMu sync.Mutex` dedicado, adquirido no INÍCIO de `persist()` e liberado no fim, cobrindo a sequência `List()`+`Marshal`+`WriteFile`+`Rename` inteira — o último a adquirir o lock sempre lê o `List()` mais fresco no momento em que roda, então o resultado final em disco sempre reflete o estado mais recente, não o mais lento a terminar.
   - `[→ review/patterns.md#persistência-em-disco-fora-da-seção-crítica]` `[→ security.md#SEC-105]`

### Non-bloqueantes
- suggestion (observability, if-minor): `persist()`/`load()` (`devices/store.go:54-91`) engolem TODOS os erros de I/O/marshal silenciosamente (`return` sem log) — inconsistente com o resto do hub (`notifier.go` loga falha de push com `slog.Warn`). Um disco cheio ou erro de permissão pararia a persistência sem nenhum sinal operacional. Vale ao menos um `slog.Warn` nos `return` de erro.
- suggestion (testability): nenhum teste novo de `devices/store_test.go` exercita `Upsert`/`Remove` concorrentes (`TestPersistenceRoundTrip`/`TestPersistenceRemove` são sequenciais) — um teste com N goroutines concorrentes + `-race` + verificação de que o conteúdo final em disco bate com o estado final em memória cobriria o achado #4 acima (e falharia hoje, sem o `persistMu`).
- suggestion (correctness, non-blocking, baixa severidade): `Launcher.Resolve` (`launcher.go:414-419`) força o estado para `done` incondicionalmente, sem checar se o estado atual ainda é `needsYou` — uma corrida estreita entre o swipe "Concluir" no app e uma transição legítima concorrente (ex.: novo prompt enviado no terminal levando a `running` quase no mesmo instante) pode fazer o `Resolve` sobrescrever esse `running` de volta para `done`. Considerar `Resolve` ser no-op (ou retornar um erro de conflito) se o estado atual não for `needsYou` no momento da chamada.
- nitpick (doc): `docs/09-configurar-hooks.md` está desatualizado — não documenta os campos `pane`/`tmux_socket`/`machine` que o forwarder de hook agora envia, nem a distinção Notification ociosa vs. bloqueio real. Vale atualizar junto, para quem configurar hooks numa máquina nova ter referência correta.

### Confirmações positivas (respondendo aos pontos específicos pedidos)
- **Semântica ocioso→done não engole pedido real (ponto d):** confirmado correto. `notificationBlocks` (`hooks.go:109-116`) só classifica como NÃO-bloqueio as substrings "waiting for your input"/"waiting for input"; qualquer outra mensagem — incluindo "Claude needs your permission...", qualquer prompt de aprovação de plano, e qualquer mensagem customizada/desconhecida — cai no default (bloqueia, vira `needs_you`). O default conservador é explícito e documentado no próprio comentário da função, e as duas mensagens reais do Claude Code (permissão vs. ociosidade) são tratadas de forma distinta e testada (`TestHookNotificationPermissionBlocks`, `TestHookIdleNotificationBecomesDone`, `TestHookIdleDoesNotResurrectDone`).
- **Edge cases de correlação pane→estado / roteamento no app (ponto c):** "sessão ainda não carregada" é tratado corretamente — `resolveDeepLink()` é re-executado via `.onChange(of: model.sessions)` (`SessionListView.swift:364`), então um push tocado antes do carregamento inicial da REST terminar resolve corretamente assim que as sessões chegam, sem ficar preso. "pane ausente" também é seguro — `tmuxTarget` retorna `nil` para pane vazio, e todo o dedup usa `compactMap`/`?? false`, então uma sessão sem pane nunca casa falsamente com nenhum Set de dedup. O dedup entre "Ao vivo"/"Precisa de você"/"Sessões" funciona corretamente dentro de UMA máquina (confirmado por trace de código) — mas quebra entre máquinas diferentes, ver bloqueante #3.
- **Concorrência da persistência (ponto b):** confirmado com defeito real — ver bloqueante #4. O padrão tmp+rename em si está correto (atômico contra o processo morrer no meio); o problema é a ausência de serialização ENTRE chamadas concorrentes de `persist()`, uma classe de bug diferente da que `go test -race` consegue pegar.
- **Raciocínio anti-spam do notifier (ponto a):** o comentário que afirma "um push por tarefa concluída, não por turno" está incorreto para sessões usadas interativamente — ver bloqueante #1 com o trace completo do ciclo `UserPromptSubmit→running→Stop→done` repetido.

### Bloqueantes
1. **issue (correctness/concurrency, blocking):** corrida entre o hook (`HookHandler`/`EnsureRegistered`, `hooks.go:113-125` + `engine.go:117-146`) e o `Runner` do próprio hub (`ensureRunning`, `engine.go:86-107`) para o MESMO `session_id`, quando o hub lança um `claude -p` LOCAL (o hook `~/.cutuque/hook.sh` dispara para "TODA sessão claude do Mac, inclusive as -p lançadas pelo hub" — premissa confirmada pelo próprio pedido de revisão). Quando o `session_started` REAL chega depois, `ensureRunning` cai no branch `exists=true` (linha 103-106) — que só ajusta `State`/`Pane`, NUNCA reverte `External` para `false` nem corrige `Title`/`Machine`/`Agent`. Nenhum teste cobre esta ordem de eventos (`engine_test.go`/`hooks_test.go` só testam cada caminho isoladamente).
   - **Por que importa:** quebra o invariante de segurança do docs/03 ("nunca aprovar às cegas") pelo lado oposto — a sessão não é aprovada às cegas, mas fica IMPOSSÍVEL de aprovar, travando um agente que pode estar no meio de uma operação.
   - **Sugestão concreta:** dar ao `Runner`/`Launcher` uma forma de "reclamar" a sessão como hub-owned de forma que sempre vença, independente da ordem: ou (a) adicionar uma marca de origem ao `event.Event` (ex. `Source: "runner"`) e fazer `ensureRunning` SEMPRE corrigir `Title`/`Machine`/`Agent`/`External=false` quando `Source=="runner"`, mesmo se a sessão já existir; ou (b) separar `ensureRunning` (só chamado pelo Launcher) de um `ensureRunningFromHook` dedicado, e nunca deixar o caminho do hook sobrescrever campos depois que o Launcher "reclamar" a sessão.

2. **issue (correctness/concurrency, blocking — 4ª ocorrência do padrão `reivindicação-não-atômica-de-recurso-compartilhado`):** `Engine.EnsureRegistered` (`engine.go:117-146`) faz `Get` + `Dismissed` + `Add` como TRÊS aquisições de lock separadas do Registry, não uma seção crítica única — apesar de o Registry já expor `AddIfAbsent` (atômico, `registry.go:86-96`) para exatamente este propósito, já usado por `Launcher.Adopt` (`launcher.go:377`). Isso reabre a mesma classe de bug catalogada 3x no projeto (a última em `Adopt`, já corrigida com `AddIfAbsent`) por um caminho NOVO que essa correção não cobriu. Colisão concreta e demonstrável: `Launcher.Adopt` (`launcher.go:350-388`) chama `l.reg.Undismiss(id)` (linha 361) e só DEPOIS `l.reg.AddIfAbsent(s)` (linha 377) — duas operações de lock separadas, com a montagem do `session.Session{}` no meio. Se um hook concorrente para o MESMO `id` (plausível: a usuária está literalmente reabrindo/continuando uma sessão do Claude que ainda está viva no tmux, gerando eventos de hook, exatamente o cenário que "Continuar sessão do Mac" existe para servir) executar `EnsureRegistered` na janela entre o `Undismiss` e o `AddIfAbsent` do Adopt, o hook vence a corrida: `Get(id)`→false, `Dismissed(id)`→false (acabou de ser limpo pelo Undismiss), `Add(...)` insere uma versão hook-flavored (título de `niceNameFromCwd`, sem o `title` que a usuária escolheu ao adotar). Quando o `Adopt` chega no seu próprio `AddIfAbsent(s)`, encontra o id já existente (`added=false`) e **pula o `importTranscript`** (linha 378-380: `if !added { return existing, nil }`) — a sessão reaparece sem o histórico importado, e com metadados errados. É o mesmo efeito visível para a usuária (histórico "sumido") que motivou a correção anterior deste padrão, só que pela porta que ficou destravada.
   - **Por que importa:** o padrão já mordeu o projeto 3x antes; desta vez reabre um bug que JÁ tinha sido fechado (Adopt/importTranscript) através de um caminho de entrada novo introduzido por esta mesma feature.
   - **Sugestão concreta:** trocar o trio `Get`/`Dismissed`/`Add` de `EnsureRegistered` por uma única operação atômica no Registry, ex. `func (r *Registry) EnsureRegisteredIfNotDismissed(s session.Session) (added bool)` que faz a checagem de existência + dismissed + insert sob o MESMO lock (mesmo padrão de `AddIfAbsent`, só que também consultando `dismissed` dentro da seção crítica).
   - `[→ review/patterns.md#reivindicação-não-atômica-de-recurso-compartilhado]`
   - **Status: RESOLVIDO nesta leva** — `EnsureRegistered` (`engine.go:124-149`) agora usa `Registry.AddIfAllowed` (`registry.go:84-98`), que faz `dismissed`+existência+insert sob o MESMO lock. Confirmado via leitura de código; ver, porém, o achado #2 da entrada acima (`ensureRunning` reabre o mesmo padrão por um caminho irmão).

3. **issue (correctness, blocking):** dedup por pane no app (`SessionListView.swift`) não detecta um pane REUTILIZADO por uma sessão diferente. `Registry.SetPane`/`ensureRunning` gravam `Session.Pane` uma vez e nunca o limpam (`registry.go:142-156`: no-op se `pane==""`); não existe nenhuma reconciliação no hub que detecte "este pane agora pertence a outra sessão" (confirmado: nenhum mecanismo de cleanup/TTL/GC no hub — `grep` não encontrou nada). Cenário: sessão A trava em `needs_you` (ex.: processo morto com `kill -9`/Ctrl+C antes de o `Stop` hook disparar — comum quando a usuária desiste de um pedido de permissão assustador) e permanece nesse estado para sempre no registry, com `Pane = "sock\t%3"`. A usuária inicia um NOVO `claude` na MESMA pane %3 (fluxo normal de reuso de terminal) → hook registra a sessão B com o MESMO pane. Na lista (`SessionListView.swift:196-209`), `needsYou` (linha 197) não filtra por pane — a entrada STALE de A continua aparecendo em "Precisa de você" indefinidamente.
