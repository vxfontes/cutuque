## superfície-secundária-não-replica-o-gate-de-autorização-da-superfície-primária
**Categoria:** security | design
**Frequência:** 2x (as duas ocorrências nesta mesma leva) | **Primeira vez:** 2026-07-10 | **Última:** 2026-07-10
**Descrição:** quando uma ação sensível nova é adicionada (aqui: responder/cancelar uma pergunta de seleção `AskUserQuestion`), o guard que decide QUANDO essa ação é segura de expor (aqui: `!isExternal` — sessão lançada pelo app vs. adotada de hook/terminal, que o hub não controla via canal de controle) é implementado corretamente na superfície PRIMÁRIA (`SessionDetailView.swift`, iOS) mas não é propagado às superfícies SECUNDÁRIAS que expõem a MESMA ação por um canal diferente. Duas ocorrências confirmadas na mesma revisão: (1) o relógio (`WatchConnector`/`WatchRootView`/`PhoneWatchRelay`) — o payload da ponte WatchConnectivity nem carrega o campo `isExternal`, só o proxy pré-existente `hasPane`, e o novo branch `hasQuestions` não reaplica nenhum gate, ao contrário do branch de Aprovar/Negar que já tinha `!session.hasPane`; (2) a notificação push (`hub/internal/notifier/notifier.go#buildPush` + `app/CutuqueApp/PushManager.swift`, nenhum dos dois tocado por esta leva) — a categoria `NEEDS_YOU` sempre expõe as quick actions `APPROVE_ACTION`/`DENY_ACTION`, sem distinguir se o pending é uma pergunta de seleção ou uma permissão comum, oferecendo um caminho de UM TOQUE, direto da tela de bloqueio, para acionar `/approve` sobre um pending que na verdade é um `AskUserQuestion` — o caminho de reprodução mais direto encontrado para o SEC-111 já catalogado (aberto).
**Recomendação:** ao adicionar uma ação sensível nova que só é segura sob uma condição já modelada em UMA superfície (ex.: `Session.isExternal`), fazer um inventário explícito de TODAS as superfícies que hoje expõem ações irmãs da mesma sessão (lista viva atual: tela de detalhe iOS, lista iOS com swipe, relógio via WatchConnectivity, notificação push com quick actions) antes de considerar a mudança completa — e propagar o MESMO campo de gate (não um proxy correlacionado, como `hasPane`) para cada uma. Se este padrão aparecer de novo (3ª ocorrência, fora desta leva), promover para uma checklist formal de "matriz de superfícies x ações" no processo de review.

## campo-de-desambiguação-adicionado-mas-nunca-lido
**Categoria:** correctness | design
**Frequência:** 1x (candidato — critério de promoção é 2x) | **Primeira vez:** 2026-07-10
**Descrição:** um campo é adicionado a uma struct especificamente para permitir uma validação nova (o próprio comentário do campo descreve o propósito de checagem), a leitura/gravação do campo é implementada, mas o CONSUMO (o `if` que efetivamente usa o campo para decidir/rejeitar) nunca é escrito — o campo fica "morto" (gravado, nunca lido), e o código segue em frente assumindo, sem checar, que o caso que o campo existiria para distinguir nunca vai acontecer. Visto em `hub/internal/launcher/launcher.go`: `pending.toolName` foi adicionado com o comentário explícito "toolName distingue uma pergunta de seleção (AskUserQuestion, respondida via Answer) de um pedido comum de permissão (respondido via Approve/Deny)", preenchido em `launchApplier.Apply`, mas `Launcher.Answer`/`respond` nunca leem `p.toolName` antes de agir — `Answer` constrói e envia um "allow" incondicional para QUALQUER pending, mesmo quando não é um `AskUserQuestion` (confirmado por PoC, ver `security.md#SEC-111`).
**Recomendação:** quando um campo é adicionado com um comentário que descreve seu propósito como "distinguir X de Y para decidir Z", tratar a AUSÊNCIA do `if` correspondente como um sinal de revisão automático — grep pelo nome do campo no arquivo antes de aprovar; se o único uso for a atribuição (nunca uma leitura em um `if`/`switch`), o código não fez o que o comentário promete. Se este padrão aparecer de novo, promover para recomendação definitiva com uma regra de lint/checklist explícita de review.
`[→ security.md#SEC-111]`

## reivindicação-não-atômica-de-recurso-compartilhado
**Categoria:** correctness | concurrency
**Frequência:** 5x | **Primeira vez:** anterior a 2026-07-03 (`Handle.Close`, `Launcher.respond` — histórico completo em `hub/review/patterns.md#reivindicação-não-atômica-de-recurso-compartilhado`) | **Última:** 2026-07-03 (`Engine.ensureRunning`)
**Descrição:** Um recurso compartilhado (registry, handle de processo, canal de resposta pendente) é "reivindicado" por check-then-act em vez de uma operação atômica única — duas goroutines/requests concorrentes passam ambas pelo `check` antes de qualquer uma fazer o `act`, e a segunda pisa ou duplica o trabalho da primeira. Histórico:
1. `Handle.Close` — corrigido com `sync.Once`.
2. `Launcher.respond` — corrigido com reivindicação atômica do pending.
3. `Launcher.Adopt` (`reg.Get` + `reg.Add` + `importTranscript`) — corrigido com `Registry.AddIfAbsent` (lookup+insert sob um único lock); `Adopt` só roda `importTranscript` quando `added == true`. Confirmado RESOLVIDO (`launcher.go:377`, `l.reg.AddIfAbsent(s)` já em uso).
4. `Engine.EnsureRegistered` (`hub/internal/engine/engine.go`) fazia `Get` + `Dismissed` + `Add` como três aquisições de lock separadas do Registry, reabrindo uma janela de corrida com o próprio `Launcher.Adopt` (`Undismiss` seguido de `AddIfAbsent` como duas operações separadas). Confirmado RESOLVIDO nesta leva: `EnsureRegistered` agora usa `Registry.AddIfAllowed` (`registry.go:84-98`), que faz `dismissed`+existência+insert sob o MESMO lock.
5. **NOVO (2026-07-03):** `Engine.ensureRunning` (`hub/internal/engine/engine.go:86-102`) — o caminho IRMÃO de `EnsureRegistered` que também cria sessão do zero (usado pelo `Runner` e pelo dispatch de hook `SessionStart`/`UserPromptSubmit` via `Apply`) NÃO foi migrado junto: seu branch de criação continua fazendo `Get` seguido de `Add` bruto como duas aquisições de lock separadas, em vez de `AddIfAbsent` (já disponível, já usado pela correção da ocorrência #3/#4). Cenário: hook e Runner recebem `SessionStarted` de uma sessão nunca vista por nenhum dos dois quase simultaneamente (a mesma premissa do SEC-103 original); se um `Get` retornar `exists=false` e, antes de seu `Add` bruto rodar, o outro lado já tiver inserido E avançado o estado (ex.: para `needs_you` com `PendingPrompt`), o `Add` bruto SUBSTITUI o registro inteiro por um `session.Session{}` novo em `StateRunning`, apagando silenciosamente o `needs_you`/`PendingPrompt`/`Pane` que já existiam — pior que o SEC-103 original porque é uma reversão SILENCIOSA (sem badge de travado) enquanto o processo real continua bloqueado esperando resposta.
**Recomendação:** sempre que um novo caminho de escrita para um recurso compartilhado do Registry for criado (novo handler HTTP, novo hook, novo evento), usar a mesma primitiva atômica já validada (`AddIfAbsent`/`AddIfAllowed`) ou estendê-la, em vez de compor múltiplas chamadas de lock separadas. Regra prática de review: quando uma correção deste padrão for aplicada a UM caminho de criação de sessão, procurar explicitamente por OUTROS caminhos que também criam sessão do zero (grep por `reg.Add(` cru e por `reg.Get` seguido de `reg.Add` no mesmo método) antes de considerar o padrão fechado — este padrão já reapareceu 2x (ocorrência #4 reabrindo o que a #3 tinha fechado; ocorrência #5 reabrindo o que a #4 fechou) por um caminho irmão não coberto pela correção anterior. Fix concreto pendente: `ensureRunning` deveria usar `e.reg.AddIfAbsent(novaSessao)`; se `added==false`, cair no mesmo bloco de reconciliação (`Reclaim`/`UpdateState`/`SetPane`) já existente para o caso "já existia".

## ssh-remote-script-sem-login-shell
**Categoria:** design | performance/reliability
**Frequência:** 7x | **Primeira vez:** 2026-07-03 (`claudecode/discover.go`) | **Última:** 2026-07-10 (`opencode/discover.go`, `opencode/transcript.go`)
**Descrição:** Os adapters que rodam scripts Python numa máquina remota via SSH montam o comando remoto como `ssh <sshBaseOpts> -- <dest> "python3 -"` (ou `"python3 - "+SingleQuote(id)` no caso do Transcript), sem envolver em `bash -lc` (login shell) — diferente do padrão usado para rodar o próprio agente (`remoteClaudeCommand`/`remoteCodexCommand`/`remoteOCCommand`, que usam login shell explicitamente via `bash -lc 'exec "$0" "$@"'`). Sem login shell, o comando remoto roda no shell não-interativo default do `sshd` (geralmente `sh`, não necessariamente lendo `.bashrc`/`.zshrc`/`.profile`), o que pode não ter o mesmo `PATH` de uma sessão interativa — um `python3` instalado via um gerenciador de versão (pyenv/asdf/homebrew em PATH customizado no `.zshrc`) pode não ser encontrado, fazendo `Discover`/`Live`/`Transcript` falharem silenciosamente (erro tratado como `ErrDiscoverFailed`/falha do `importTranscript`) em máquinas com esse tipo de setup, mesmo que o agente em si funcione (porque esse SIM usa login shell). Histórico completo de ocorrências: `claudecode/discover.go`, `claudecode/live.go`, `claudecode/transcript.go` (2026-07-03, 3x) → `codex/discover.go`, `codex/transcript.go` (2026-07-06, +2x, não contabilizado na entrada anterior) → `opencode/discover.go`, `opencode/transcript.go` (2026-07-10, +2x — este é o TERCEIRO adapter consecutivo a replicar exatamente a mesma decisão de design, cada um via copy-paste independente em vez de um helper compartilhado entre packages).
**Recomendação:** Ajustar cada helper (`runDiscoverScript`/`runDiscover`/`runTranscript`, hoje duplicado em `claudecode/`, `codex/` e `opencode/`) para envolver o `python3 -` em `bash -lc` também, na mesma linha do que já é feito para o próprio agente. Como o padrão já se replicou em TRÊS packages independentes sem correção, considerar extrair um helper único em `internal/adapter/agent` (ex.: `agent.RunRemotePython(ctx, dest, prog, script string, args ...string) ([]byte, error)`) que todo adapter novo (um eventual 4º agente) reusaria por padrão — em vez de continuar confiando que cada novo adapter vai lembrar de aplicar a mesma correção manualmente. Continua não-bloqueante (falha é tratada como erro, não como corrupção/injeção), mas o custo de correção só cresce a cada adapter novo que copia o padrão atual.

## shell-command-por-string-reparseada-em-shell-aninhado
**Categoria:** security
**Frequência:** 1x (candidato — critério de promoção é 2x) | **Primeira vez:** 2026-07-03
**Descrição:** construir um comando remoto concatenando argumentos numa única string e depois envolvê-la em UM NÍVEL de single-quote para passar a um `bash -lc` — o valor quotado vira o argumento `-c` de um shell ANINHADO, que o reparseia como sintaxe de shell, anulando o escape do nível externo para qualquer input que contenha `;`, `&&`, `$()`, etc. Visto em `hub/internal/adapter/claudecode/target.go` (`remoteClaudeCommand`), causa raiz do SEC-101 (corrigido). Confirmado em três levas seguintes (transcript.go/live.go, tmux.go, e agora o payload composto de pane em hooks.go) que o padrão NÃO se repetiu — todos usam um único nível de shell.
**Recomendação:** se reaparecer em outro adapter (ex.: um futuro `DockerTarget`/`K8sTarget`), promover a entrada completa aqui com recomendação: usar `bash -lc 'exec "$0" "$@"' arg0 arg1 ...` (cada argv quotado individualmente) em vez de string única reparseada.

## chave-composta-sem-dimensão-de-desambiguação
**Categoria:** correctness | design
**Frequência:** 1x (candidato — critério de promoção é 2x) | **Primeira vez:** 2026-07-03
**Descrição:** uma chave composta usada para correlacionar recursos (ex.: `"<socket>\t<pane>"` do alvo tmux, montada em `hookPayload.paneTarget()`, `hub/internal/server/hooks.go:32-43`) omite uma dimensão de desambiguação que existe no domínio (`machine`) mas não está representada em NENHUM lugar da string nem checada separadamente nos pontos de comparação. Como o Cutuque suporta nativamente múltiplas máquinas (`Launcher.targets`/`SSHTarget`/`LocalTarget`) e tmux usa defaults previsíveis (socket "default", pane "%0"), duas máquinas diferentes rodando cada uma uma sessão tmux default colidem na MESMA string de pane com boa probabilidade na prática — não é um edge case exótico. Efeito confirmado em dois lugares que comparam essa chave sem `machine`: `Registry.SetPane` (hub, `registry.go:189-230`, loop de eviction) e `SessionListView.swift` (app, `livePaneIDs`/`needsYouPaneIDs`/`others`/`liveState`, todos `Set<String>`/comparação de `tmuxTarget` sem `machine`).
**Recomendação:** se este padrão aparecer de novo em outra chave composta do projeto, promover para recomendação definitiva: incluir TODAS as dimensões de desambiguação relevantes do domínio na própria chave (ex. `"<machine>\t<socket>\t<pane>"`), em vez de confiar que todo código futuro que compare a chave vai lembrar de checar a dimensão extra separadamente — a ambiguidade embutida numa string é fácil de esquecer de tratar em um novo call site.

## persistência-em-disco-fora-da-seção-crítica
**Categoria:** correctness | concurrency
**Frequência:** 1x (candidato — critério de promoção é 2x) | **Primeira vez:** 2026-07-03
**Descrição:** um efeito colateral de persistência em disco (write-tmp + rename atômico) é disparado FORA do lock que protege a mutação em memória correspondente, e nada serializa esse efeito colateral contra chamadas concorrentes de si mesmo. Visto em `hub/internal/devices/store.go` (`persist()`, chamado fora do `sync.RWMutex` em `Upsert`/`Remove`): o padrão tmp+rename é atômico contra o processo morrer no meio de UMA chamada, mas não é serializado entre DUAS chamadas concorrentes — a que TERMINA por último vence, independente de qual mutação era logicamente mais recente. Esta classe de corrida é invisível a `go test -race` (é uma corrida de I/O de arquivo do SO entre chamadas de `os.WriteFile`/`os.Rename`, não uma corrida de memória gerenciada pelo runtime Go).
**Recomendação:** se este padrão aparecer de novo em outro componente que persiste estado em disco a partir de múltiplos writers concorrentes, promover para recomendação definitiva: um mutex dedicado à operação de persistência (separado ou não do mutex de memória) cobrindo a sequência inteira de leitura-do-estado-atual + serialização + write + rename, garantindo que o último a persistir sempre leia o estado mais fresco no momento em que roda. Lembrar explicitamente que `go test -race` NÃO pega esta classe de bug — vale um teste dedicado com N goroutines concorrentes chamando a operação de persistência e verificando o conteúdo final em disco contra o estado final em memória.
