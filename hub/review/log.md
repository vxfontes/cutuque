### Bloqueantes
- issue (correctness/concurrency, blocking): `claudecode.Handle.Close()` (`internal/adapter/claudecode/target.go:28-39`) não é seguro contra chamada concorrente por duas goroutines quando apoiado num processo real (`closer = cmd.Wait`, setado em `target.go:150`). Em `launcher.Launch` (`internal/launcher/launcher.go:104-121`), no caminho de timeout, a goroutine do `select` chama `handle.Close()` **ao mesmo tempo** que a goroutine de fundo (que roda `runner.Run`) também chama `handle.Close()` assim que o `Close()` da primeira fecha `Stdout` e derruba o loop de leitura — as duas chamadas concorrentes a `h.closer()` (`cmd.Wait()`) disparam uma **data race real dentro do próprio `os/exec`**, confirmada experimentalmente com `go test -race` (reproduzido num teste temporário chamando `newLocalCommand` + duas goroutines fazendo `h.Close()`, removido do repo depois — `git status` limpo). Por que importa: nenhum teste do repo pega isso porque os fakes de teste (`scriptTarget`, `io.Pipe`) nunca setam `closer`, então `TestLaunchTimeout` passa mesmo com o bug presente — o baseline `go test -race` fica verde apesar do bug ser 100% real e reproduzível em produção toda vez que um `Launch` real estoura o timeout. Sob `-race` em produção (se algum dia habilitado) isso pode até abortar o processo; sem `-race`, é undefined behavior sobre estado interno do `os/exec`.
  **Fix recomendado:** envolver a lógica de `Close()` num `sync.Once` (guardando também o erro), para que não importa quantas goroutines cheguem a chamar `Close()`, a operação real (incluindo `cmd.Wait()`) rode exatamente uma vez:
  ```go
  type Handle struct {
      ...
      closeOnce sync.Once
      closeErr  error
  }
  func (h *Handle) Close() error {
      h.closeOnce.Do(func() {
          if h.Stdin != nil { _ = h.Stdin.Close() }
          if h.Stdout != nil { _ = h.Stdout.Close() }
          if h.closer != nil { h.closeErr = h.closer() }
      })
      return h.closeErr
  }
  ```
  Importante: **não** trocar isso por matar o processo à força (`cmd.Process.Kill()`) no timeout — a UX atual de `NewSessionView.swift:141-145` ("o agente demorou a responder — confira a lista, a sessão pode aparecer") depende do processo poder terminar sozinho e ainda registrar a sessão via `session_started` tardio; o bug é só a falta de exclusão mútua no `Close()`, não a semântica de não matar o processo. [→ patterns.md#reivindicação-não-atômica-de-recurso-compartilhado]

- issue (correctness/security, blocking): `Launcher.respond()` (`internal/launcher/launcher.go:132-155`, usado por `Approve`/`Deny`) faz "ler para validar, soltar o lock, escrever no processo, só then limpar" sobre `l.pending`/`l.handles` — não há reivindicação atômica. Duas chamadas concorrentes para a mesma sessão (double-tap sem debounce real — o `actionInProgress` do app em `SessionDetailViewModel.runAction`, `SessionDetailView.swift:92-106`, é só uma flag de UI setada de forma otimista, não um mutex; um retry de rede do cliente ou um double-tap fisicamente rápido antes do SwiftUI re-renderizar o botão desabilitado ainda pode disparar duas `Task`s) passam AMBAS pela validação (`s.State == needs_you`, `hasPending && hasHandle`) antes que a primeira chame `clearPending`, e ambas escrevem um `control_response` (`h.WriteJSON`) para o **mesmo `request_id`** no stdin do processo real — na pior ordem, um `allow` e um `deny` para o mesmo pedido chegam ambos ao CLI. Por que importa: é exatamente a feature-título da Fase 3 (aprovar/negar de longe) tendo uma janela de corrida sem teste de regressão que a cubra (`TestApproveWritesExactControlResponseAndResumes` só testa uma chamada sequencial). O comentário do próprio pacote declara a invariante "aprovar/negar exige que a sessão esteja mesmo em needs_you" (`launcher.go:9-10`), mas a checagem não é atômica com a ação.
  **Fix recomendado:** mover a reivindicação de `pending`/`handle` para dentro da MESMA seção crítica que decide se a ação prossegue — ex.: `delete` do mapa já dentro do lock que fez o `lookup`, e só prosseguir se `hasPending && hasHandle` **no momento do delete**:
  ```go
  l.mu.Lock()
  p, hasPending := l.pending[id]
  h, hasHandle := l.handles[id]
  if hasPending && hasHandle {
      delete(l.pending, id) // reivindica atomicamente; a segunda chamada não encontra mais nada
  }
  l.mu.Unlock()
  if !hasPending || !hasHandle {
      return ErrStaleState
  }
  ```
  Adicionar um teste que dispare `Approve`/`Deny` concorrentes (via `sync.WaitGroup`, `go test -race`) e assert que só UM `control_response` chega ao processo. [→ patterns.md#reivindicação-não-atômica-de-recurso-compartilhado]

- issue (security, blocking): `LocalTarget.Start` (`internal/adapter/claudecode/target.go:134-152`) cria o `exec.Cmd` sem definir `cmd.Env`, então o processo `claude` herda **todo** o ambiente do hub — inclusive `CUTUQUE_TOKEN`, o único segredo que protege toda a API do hub. O próprio docs/10 (armadilha #2) documenta que comandos "seguros" (ex. `echo`) rodam sem passar pelo `control_request` — sem chance de veto do hub — então um `env`/`printenv` induzido (prompt injection de conteúdo externo que o agente processe, ou simplesmente o usuário pedindo "mostra as env vars") vaza o token para o `output_chunk`, ou pode ser exfiltrado por um comando de rede subsequente, sem nunca passar pelo gate de aprovação. Docs/05 já declara que segredos futuros (APNs `.p8`) "ficam só no hub" — o mesmo vazamento se aplicará a eles se também virarem env var do processo hub. [→ security.md#SEC-006]
  **Fix recomendado:** montar um `cmd.Env` explícito (allowlist mínima: `PATH`, `HOME`, locale, credencial de API do próprio Claude Code se for via env) em vez de herdar `os.Environ()`. Teste: setar uma env var sentinela no processo de teste e assertar que ela NÃO aparece no ambiente do processo filho.

Não verificado como alcançável contra o CLI real (docs/10 não testou tool use paralelo) — vale um follow-up de probe antes de decidir se compensa transformar `pending` numa fila.
- note (design, non-blocking): a ausência de graceful shutdown (`cmd/hub/main.go`, já registrado no achado da Fase 1) ganha um efeito colateral novo na Fase 3: como `LaunchHandler` sempre chama `Launch(context.Background(), ...)` (`launch.go:70`, deliberado — comentário explica que a sessão precisa sobreviver à resposta HTTP), não existe NENHUM caminho de cancelamento do processo do agente hoje, nem por timeout de request nem por shutdown do hub. Na prática, se o hub morrer, o processo filho provavelmente recebe EPIPE ao tentar escrever no stdout (o pipe fecha junto com o hub) e tende a sair sozinho — não é um zumbi eterno garantido — mas não há reconciliação na subida (já prevista em docs/05:33-34) para o hub redescobrir/reconectar a esse processo depois, então qualquer trabalho em andamento nesse restart é silenciosamente perdido sem registro do que aconteceu. Não bloqueio (é dívida já rastreada), mas o escopo do risco mudou: antes era só estado em memória; agora é um processo de verdade com custo de API em andamento.

**Escopo:** Verificação dos 2 bloqueantes da Fase 1 (auth bypass, keepalive WS) + revisão completa de `internal/event`, `internal/engine`, `internal/adapter/claudecode` (parser, runner, target, fixtures), `internal/registry/output.go`, `internal/server/{hooks,output,ws,server}.go`, `internal/integration`, e primeira revisão de todo `app/CutuqueApp/*.swift`. Rodado `go vet ./...` e `go test -race ./...` no hub (ambos limpos). Sem test target no app iOS — nada a rodar do lado Swift.

### Verificação dos bloqueantes da Fase 1
- SEC-001 (auth bypass com token vazio): **corrigido corretamente** — fail-fast em `cmd/hub/main.go:16-19` (`os.Exit(1)` se `prod`+token vazio) e defesa em profundidade em `auth.go` (`validToken` nunca aceita `configured == ""`, comparação constant-time via `subtle.ConstantTimeCompare`, cobrindo também SEC-002 de brinde). Teste novo cobre o comportamento, não só o valor de config.
- Bloqueante #2 (WS sem timeout/keepalive): **corrigido corretamente** — `ws.go` adiciona ping periódico (30s) com timeout de escrita (10s) via `c.Ping(pctx)`, serializado no mesmo `select` que as escritas normais (sem risco de escrita concorrente na conexão). `TestWSSurvivesPingTicks` prova que o heartbeat não quebra o stream normal. Não há teste que force uma conexão realmente pendurada (TCP sem FIN/RST) para provar que o timeout de fato libera o recurso — aceitável, é difícil de simular sem mexer no transporte; registrado como lacuna de teste, não bloqueante.

### Bloqueantes
- issue (design, blocking): `claudecode.Runner.registerSession` (`internal/adapter/claudecode/runner.go:88-99`) escreve direto no Registry (`r.reg.Add(...)`) para criar a sessão com metadados (machine/agent/title), **contornando o Engine**. Isso viola o invariante declarado no próprio `engine.go:3-4` ("[Engine] é a única peça que escreve o estado das sessões"). Consequência concreta: `Engine.ensureRunning`'s ramo de criação (`engine.go:67-77`) é código morto no fluxo real — hoje só é exercitado pelo teste unitário `TestApplySessionStartedCreatesRunningSession`, nunca pelo Runner de verdade, porque quando `Engine.Apply` processa o `session_started`, a sessão já foi criada por fora. Por que importa: (a) todo adapter futuro (Codex, OpenCode, citados no doc de `event.go`) vai precisar do mesmo bypass, já que `event.Event` não carrega Machine/Agent/Title — o contrato está anêmico demais para o que o Runner precisa fazer; (b) qualquer validação central que a Fase 3 queira colocar no Engine (ex.: idempotência, auditoria, rate limiting de criação) pode ser furada por quem escrever direto no Registry; (c) é a 3ª ocorrência do padrão `invariante-declarada-mas-não-garantida` já catalogado. [→ patterns.md#invariante-declarada-mas-não-garantida]
  **Fix recomendado:** estender `event.Event` com os campos de metadados necessários (ou criar um evento dedicado tipo `SessionStarted{Machine, Agent, Title}`), e mover a criação completa da sessão para dentro de `Engine` (ex.: `Engine.ensureRunning` passa a receber e usar esses campos). `Runner` deixa de importar/tocar `*registry.Registry` diretamente — só fala com `Engine`.

- issue (correctness, blocking) `app/CutuqueApp/SessionListView.swift:32-50` e `SessionDetailView.swift:31-53`: ambos os `ViewModel.startLiveUpdates()` criam `Task { [weak self] in guard let self else { return }; for await message in self.api.liveUpdates() { ... } }`. O `guard let self` está **fora** do `for await`, então a partir do momento em que desembrulha com sucesso, a closure retém `self` fortemente por toda a duração do loop (que só termina quando a Task é cancelada). Como o próprio `self` guarda essa Task em `liveTask` (`self -> liveTask -> closure -> self`), isso é um ciclo de retenção genuíno: o ViewModel só é liberado se algo *externo ao ciclo* chamar `.cancel()` (hoje, só `onDisappear`). Por que importa: se por qualquer razão `onDisappear` não disparar (troca de tab, sheet, mudança futura de navegação), o ViewModel, a Task e a conexão WebSocket aberta com o hub vazam para sempre — é exatamente a mesma classe de bug já achada e corrigida do lado do hub na Fase 1 (recurso de longa duração sem cancelamento garantido), agora do lado cliente. É a 2ª ocorrência do mesmo antipadrão dentro desta mesma revisão (nos dois ViewModels).
  **Fix recomendado:** mover o unwrap de `self` para dentro do loop, reavaliado a cada iteração:
  ```swift
  liveTask = Task { [weak self] in
      guard let stream = self?.api.liveUpdates() else { return }
      for await message in stream {
          guard let self else { break }
          // ... usa self ...
      }
  }
  ```
  Assim `self` só fica retido durante o processamento de cada mensagem, nunca durante toda a vida do stream. [→ patterns.md#weak-self-fora-do-loop-em-task-de-stream]

- issue (security/perf, blocking): `HookHandler` (`internal/server/hooks.go:28-29`) faz `json.NewDecoder(r.Body).Decode(&p)` sem `http.MaxBytesReader`/limite de tamanho. É o único endpoint POST do hub e já está em produção-elegível (roteado em `server.go:28`). Qualquer chamador com o token válido (inclusive um token vazado via SEC-003) pode mandar um body arbitrariamente grande e forçar o hub a bufferizar tudo em memória — DoS trivial contra o único processo do usuário. **Fix recomendado:** `r.Body = http.MaxBytesReader(w, r.Body, 64*1024)` antes do `Decode`, respondendo 400/413 em overflow (ver `security.md#SEC-004`).

### Non-bloqueantes
- suggestion (perf, non-blocking): `claudecode.Runner.Run` (`runner.go:43,48`) lê com `bufio.NewReader(rc).ReadBytes('\n')`, escolha correta para não quebrar em linhas grandes (testado até 200KB em `TestRunnerHandlesLongLines`, ao contrário do `bufio.Scanner` que tem limite fixo de 64KB) — mas como efeito colateral não há **nenhum** teto. Uma linha patológica (ex.: um `tool_result` com um blob base64 grande, sem quebra de linha) cresce a memória do hub sem limite. Ainda não está no caminho quente (Runner não está plugado no `main.go`/roteador ainda), mas vale blindar antes de conectar. **Fix recomendado:** envolver a leitura com um teto de segurança (ex.: `io.LimitReader` por linha, ou contar bytes acumulados e abortar/truncar acima de ~10-20MB), preservando o comportamento já testado até 200KB.

Uma sessão longa/verbosa mantida aberta acumula memória e re-renderiza uma lista cada vez maior. Sugestão: espelhar o cap de 200 (ou um valor de janela deslizante) do lado cliente também.
- suggestion (UX/correctness, non-blocking): `APIClient.liveUpdates()` (`APIClient.swift:75-86`) engole silenciosamente qualquer erro de conexão do WS dentro do loop de reconexão, sem emitir nenhum sinal de "caiu". `SessionListViewModel.hubStatus` pode ficar preso em `.online` enquanto o WS está de fato caído e tentando reconectar em background (só uma nova chamada a `/health` via pull-to-refresh corrigiria).
